# v0.3 — Payments & delivery (design & as-built notes)

> Status: **shipped** in v0.3
>
> Scope from the [roadmap](../README.md#roadmap): *Stripe import
> (customers/products) + "Pay now" links, recurring invoices, email delivery.*
> Written in the same shape as the
> [v0.2 design](./workspace-mode-design.md); the "as built" notes throughout
> record where the implementation diverged from this proposal and why.

## The non-negotiables (carried forward from v0.2)

- **Instant mode stays strictly local and login-free.** Everything below lives
  under `/app/**`, `/i/**`, or a cron route — never `/`.
- **Zero env vars still yields the instant-mode-only build.** Each v0.3
  capability gates on its own env vars *on top of* `WORKSPACE_ENABLED`, and
  degrades to absent UI when unconfigured — the same pattern as
  `isWorkspaceEnabled()`.
- **No second render path.** PDF export is the browser's print pipeline and the
  preview is the document ([architecture.md](./architecture.md)). Nothing in
  v0.3 may introduce a server-side renderer to keep in sync.

## Goals / non-goals

**Goals**
- **Email delivery**: send an invoice to the client from the editor.
- **"Pay now" links**: a payment button on the shared invoice page and in the
  email.
- **Stripe import**: pull customers into saved clients, products into saved
  line items.
- **Recurring invoices**: schedules that generate drafts on cadence, optionally
  auto-sent.

**Non-goals (v0.4+)**
- Processing payments ourselves. We link out to a payment URL; no card data,
  no PSP integration on our origin.
- Stripe webhook reconciliation (auto-mark-paid). See step 7 — stretch.
- Reminder / dunning emails.
- Teams / multi-user, still.

## Feature design

### Email delivery (Resend)

`EMAIL_FROM` + `RESEND_API_KEY` (earmarked in `.env.example` since v0.2) turn
it on; `isEmailEnabled()` mirrors `isWorkspaceEnabled()` and the Send button
simply doesn't render otherwise.

**Sending is composition, not new machinery.** The send action is: mint the
share token (`shareInvoice` is already idempotent), email a link to
`/i/[token]`, and transition `draft → sent` through the existing atomic
`setInvoiceStatus` WHERE-clause guard. No new statuses; "Mark as sent" stays
for invoices delivered out of band. Re-sending a `sent` invoice is allowed,
logged, and does not transition.

**The email links to the share page; it does not attach a PDF.** An attachment
requires rendering `InvoiceDocument` server-side — exactly the second code path
the architecture forbids. The share page *is* the document, printable to PDF
by the recipient, and revocable — which an attachment never is.

**The email body stays minimal**: invoice number, amount due, due date, the
link. Line-item detail lives behind the link, not at rest in two mail
providers' archives. Copy follows `invoice.locale` via the existing `Dict`
dictionaries.

**`invoice_emails` table = audit log *and* rate limiter.** Sending email is
the first workspace action that is genuinely abusable and costlier than an
indexed lookup. v0.2 declined a rate limiter because serverless needs a shared
store — but the sends themselves are low-volume rows already going into
Postgres, so "count this user's sends in the last 24h" is one indexed query.
Per-user daily cap, env-tunable. The recipient defaults to
`invoice.client.email`, shown editable in a confirm dialog before anything
sends.

As built:

- The whole send — recipient check, cap, share-token reuse, delivery, log,
  `draft → sent` — lives in **one flow, `lib/send-invoice.ts`**, shared by
  the editor's action and the recurring generator's auto-send, so the two can
  never drift on the cap or the transition. The action contributes only what
  a request has: the session and the origin.
- **The origin comes from request headers** (`lib/origin.ts`: `host` +
  `x-forwarded-proto`, shape-checked, fail-closed), not a new base-URL env
  var. On Vercel the platform sets `host`; a value that doesn't look like a
  host refuses the send rather than emailing a poisoned link.
- The cap defaults to 20/day, tunable via `EMAIL_DAILY_CAP`. Two racing sends
  can each pass at cap−1 — the cap is anti-abuse, not accounting.
- The "confirm dialog" is an inline two-step control (`SendControls`): the
  first click only opens the editable recipient field; delivery is the
  explicit second click. Transport is a `fetch` to Resend's one endpoint —
  no SDK. Re-sending is allowed, logged, and moves nothing.
- The body is plain text and carries no line-item detail: number, amount due,
  due date, links. The full invoice lives behind the revocable link, not at
  rest in two mail providers' archives.
- **Once the provider accepts, the send is a success** — the logging and the
  `draft → sent` move get their own catch. Reporting a delivered email as a
  failure is the worse bug it resembles a fix for: the user reads "nothing was
  sent", sends again, and the client has two invoices while the first sits
  outside the audit trail and uncounted against the cap.
- The Resend call is bounded by a 10s timeout; an abort is already the
  delivery-failure path, so it needs no special casing.

### "Pay now" links

**v0.3 is paste-a-URL.** A nullable `payment_link_url` on the invoice row,
rendered as a Pay now button on `/i/[token]` and in the email
(`target="_blank" rel="noopener"`, https only). Provider-agnostic: a Stripe
Payment Link, PayPal.me, a Tikkie — whatever the user already uses.

Why not generate links via the Stripe API: creation needs `payment_links`
*write* scope, and the import key (below) is deliberately read-only. And a
generated link without webhook reconciliation still ends in a manual "mark as
paid" — so generation buys almost nothing until v0.4 does webhooks. The two
belong together, later.

**Column, not document field.** The Pay now button is a workspace affordance,
so the shared `Invoice` type stays untouched — same principle as the
`clientPicker` slot. Instant mode already has `paymentTerms` free text for
printing an IBAN or link on the sheet itself. `getSharedInvoice` grows the one
column; it remains the only owner-scoping exception and
`tenant-isolation.test.ts` keeps naming it.

As built: whatever is stored becomes an `href` served to the recipient, so
validation treats it as one (`lib/payment-link.ts`): `https:` only — which
also excludes `javascript:` — and no embedded credentials, the
`https://yourbank.com@evil.example` look-alike trick. Clearing the field
clears the column.

### Stripe import

**Whose key?** An env var is the operator's key — right for self-hosters,
wrong on the hosted multi-tenant app. So: the user pastes their own
**read-only restricted key** (Customers: Read, Products: Read, Prices: Read)
into the import UI; it is used for that import's requests, **never stored,
never logged** (narrowed to the `rk_` prefix first). No encryption
infrastructure, nothing to leak, consistent with the privacy posture.
`STRIPE_RESTRICTED_KEY` stays in `.env.example` as the self-host convenience
default that pre-fills the flow.

**Customers → saved clients.** Mapped to `Party`, then through the existing
upsert path. `clients` gains a nullable `stripe_customer_id` with a partial
unique index on `(user_id, stripe_customer_id)`, so re-imports update in place
even after a rename — name-key upsert alone would resurrect the old name as a
second row.

**Products + prices → a new `items` table** (saved line items: description,
unit price, currency, optional default VAT rate, optional Stripe ids). Stripe
has no VAT rate on a product, so imported items leave it unset and the picker
falls back to the form's default. `InvoiceForm` gains an
`itemPicker?: ReactNode` slot beside `clientPicker` — injected, never
imported, so `/` renders exactly as before.

Imports page through Stripe's list API server-side in a Server Action, capped
per run, behind `requireUserId()`.

*Open decision:* what the picker does when an item's currency differs from the
invoice's. Proposal: insert description only and leave the price for the user
— silent unit-price reuse across currencies is a wrong-invoice generator.

As built:

- The currency decision landed as proposed: a foreign-currency item inserts
  with its description and VAT default but **no price**, and the picker says
  why.
- **VAT numbers are not imported.** The proposal mapped customers to full
  `Party` objects; in practice Stripe keeps tax IDs in a separate expandable
  resource that many restricted keys can't read, and one refused `expand`
  sinks the whole list call. The field stays empty and editable — better than
  demanding a wider key.
- Preview and import are **separate actions**: the key appears only in the
  read call; the import call carries the user's selected rows, which are
  narrowed and validated like any other client payload (they took a round
  trip through the browser).
- The key is dropped from the browser's state the moment it is submitted. It
  buys exactly one read and the import never sees it, so holding it for the
  page's lifetime would be keeping a credential we have no use for.
- Reads come in **batches of 500** (5 pages × Stripe's max of 100) with a
  cursor to continue. The cursor is the point: a cap that always restarts at
  the beginning is not a batch size but a ceiling, and an account past it
  could never import the rest however often it retried. The cursor is taken
  from the last *raw* row, before mapping drops any, or resuming would skip
  everything after a dropped row.
- Customers with neither name nor email are skipped (a client needs an
  identity to be saved under); products without an expanded flat
  `default_price` (metered/tiered) are skipped likewise.
- If an imported customer's name collides with an existing client, that row is
  **adopted only if it isn't already another Stripe customer's**. Two unique
  indexes cover `clients` and a statement can only name one conflict target,
  so the name index arrives as a thrown violation — but treating every such
  violation as a plain name collision would attach this customer's id to the
  other customer's row, silently re-pointing it, and the two would then steal
  the name back and forth on every re-import. Where the name is genuinely
  contested the customer is skipped and named in the UI: the unique index says
  two clients cannot share a name, and guessing which was meant is worse than
  saying so.
- Rows are written one at a time rather than in a transaction. The Neon HTTP
  driver has no interactive transactions (the same constraint the generator
  hits), and a batch cannot express the per-row conflict resolution above. A
  partial import is safe to leave: every write is an upsert keyed on the
  Stripe id, so re-running finishes rather than duplicates.

### Recurring invoices

A new `schedules` table: owner, a template `Invoice` document, cadence
(`monthly | quarterly | yearly` — weekly invoicing is rare enough to skip),
`next_issue_date`, `payment_terms_days`, `auto_send`, `active`. No end
conditions in v0.3; `active` is the off switch.

- **Dates are computed per generation**, never copied: `issueDate` = the
  scheduled date, `dueDate` = issue + `payment_terms_days`. A literal
  `dueDate` in the template document would be stale by the second run.
- **Numbering** reuses `createInvoice`'s per-user year-scoped sequence.
- **Generation is a Vercel Cron** (daily, `vercel.json` `crons`) hitting
  `/api/cron/recurring`, guarded by `CRON_SECRET` (constant-time compare,
  404 when workspace mode is off — consistent with the flag gating).
- **Idempotency by atomic claim**, the same trick the status machine uses:
  advance `next_issue_date` in the UPDATE's WHERE clause so two overlapping
  cron runs cannot both claim a schedule; claim + insert run in one
  transaction. Missed days catch up: generate for the stored date, advance by
  cadence, repeat until the date is in the future.
- Generated invoices are **drafts** by default. `auto_send` (offered only when
  email is configured) pushes them through the step-1 send path, subject to
  the same daily cap.

As built:

- **There is no claim+insert transaction.** The proposal said "claim + insert
  run in one transaction", but the Neon HTTP driver executes single
  statements. The order chosen is claim first, insert second, with a
  best-effort compensation that rewinds the claim (guarded on the advanced
  value, so a racing run is never rewound) if the insert fails. A crash in
  that window loses one period rather than double-billing it — a period lost
  is recoverable by hand; a duplicate invoice emailed to a client is not.
- The first occurrence is **strictly after today** (`firstIssueDateAfter`):
  making an old invoice recurring schedules the future, never backfills. The
  catch-up loop then generates one invoice per *missed* period once the
  schedule is live, each dated to its own period.
- Day-of-month **clamps and stays clamped**: a schedule anchored on the 31st
  lands on Feb 28 and continues on the 28th (no anchor day is stored). Known
  simplification; invoicing on the 1st — the common case — is unaffected.
- A failed auto-send leaves the generated **draft** in place: the owner sees
  it on /app and can send by hand. Send failures never fail generation. The
  compensating rewind gets its own catch for the same reason — it runs
  *because* a write just failed, so a second failure is plausible, and an
  unguarded throw there would abandon every other due schedule in the run.
- **One schedule per source invoice**, enforced by a partial unique index on
  `(user_id, source_invoice_id)`. Hiding the button in client state was not
  enough: a reload offered it again, and a second schedule means a duplicate
  invoice — and with auto-send, a duplicate email — every period, forever.
  The index rather than a read-then-insert so two racing clicks cannot both
  pass the check, the same reasoning the invoice-number allocator uses.
- The cron route is `GET`, compares `Authorization: Bearer CRON_SECRET` in
  constant time, and **404s when unconfigured** — same story as `/app` with
  the flag off. Vercel sends the header itself once the env var exists.
- `tenant-isolation.test.ts` gained a dedicated block for `lib/recurring.ts`,
  the one module allowed to query across tenants: its scan and claim scopes
  are named (`dueSchedules`, `scheduleClaim`), and the test asserts each row
  is handed back to owner-scoped code by its own `user_id`.

## Schema changes (summary)

```
invoices        + payment_link_url  text        -- nullable; https, sender-provided

clients         + stripe_customer_id text       -- nullable
                  partial unique (user_id, stripe_customer_id)

items           id, user_id, description, unit_price, currency,
                vat_rate (nullable), stripe_product_id / stripe_price_id
                (nullable), timestamps — owner-scoped like clients

invoice_emails  id, user_id, invoice_id, to_email, sent_at, provider_id
                — audit trail and the send rate-limit counter

schedules       id, user_id, document jsonb, cadence, next_issue_date,
                payment_terms_days, auto_send, active, timestamps
```

Every new table is owner-scoped; every query filters by `user_id`;
`tenant-isolation.test.ts` grows to cover them.

## Environment / infra

- `EMAIL_FROM`, `RESEND_API_KEY` — email delivery on/off.
- `CRON_SECRET` + a `crons` entry in `vercel.json` — recurring generation.
- `STRIPE_RESTRICTED_KEY` — optional, self-host default for the import UI.
- All of it inert without `WORKSPACE_ENABLED=true`; instant mode still needs
  nothing.

## Delivery plan (incremental PRs)

Each step independently shippable; unconfigured features are invisible, not
broken.

1. ✅ **Email foundation** — Resend wiring, `isEmailEnabled()`, `invoice_emails`
   + migration, send action (share + email + `draft → sent`), confirm dialog,
   per-user daily cap. Tests: cap enforcement, transition composition,
   flag-off invisibility.
2. ✅ **Pay now** — `payment_link_url` column, editor field, button on
   `/i/[token]` and in the email. Small; independent of everything.
3. ✅ **Stripe import: customers** — paste-key flow, preview + select UI, upsert
   into `clients` with `stripe_customer_id`.
4. ✅ **Stripe import: products** — `items` table, import, `itemPicker` slot +
   picker in the editor.
5. ✅ **Recurring: schema + CRUD** — `schedules` migration, "Make recurring"
   from an existing invoice, manage page under `/app`.
6. ✅ **Recurring: generation** — cron route, `CRON_SECRET`, atomic claim +
   catch-up, `auto_send` through step 1's path.
7. **Deferred to v0.4: Stripe webhooks** — `checkout.session.completed` /
   payment events auto-marking `paid`. Needs a write-scoped key story and a
   webhook signing secret; deliberately out of the paste-a-link design above.

Dependencies: 1 before 6 (auto-send), 3 before 4 (shared import plumbing);
2 and 5 float free.

## What this preserves

- Instant mode: unchanged, local-first, zero env vars, zero network.
- The `Invoice` model, `calc.ts`, templates: reused as-is; workspace concepts
  keep arriving via injection (`clientPicker`, now `itemPicker`), never
  imports.
- The share page's capability model: the email links to it, revocation still
  works, and it stays the single named exception to owner-scoping.
